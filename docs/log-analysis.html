Local Log Analysis

Claude Code records every session as structured JSONL files on your machine. Here's what's in them, how to extract meaningful signals without sending data anywhere, and where an LLM pass earns its keep.

Where the data lives

Claude Code writes session transcripts to ~/.claude/projects/. Each project directory contains one .jsonl file per session. Every line is a typed JSON record linked by parentUuid.

~/.claude/projects/
  -Users-you-code-my-service/
    abc123def456.jsonl
    789xyz....jsonl
  -Users-you-code-other-project/
    ...

Record types

Type Key fields What it tells you
system content Full system prompt: CLAUDE.md, loaded skills, tool definitions
user message.content User prompts and hook injections — the reprompt signal lives here
assistant message.usage, message.content Per-turn token usage (input_tokens, output_tokens, cache_read_input_tokens), tool calls, thinking blocks
tool_result content, isError Whether tool calls succeeded or failed
summary summary Compaction checkpoint — context pressure forced summarisation

Two-pass analysis

The privacy-preserving architecture splits analysis into two stages. Pass 1 runs entirely locally; Pass 2 is optional and only ever sees compressed summaries — never raw code or conversation content.

Pass 1
Local heuristics
Pure computation over JSONL. No network, no LLM. Produces a ~200-token signal summary per session.
  • Token usage & cache hit ratio per turn
  • Tool error rate (isError in tool_result)
  • Reprompt count proxy: consecutive user turns with short gaps
  • Compaction events (count of summary records)
  • Ghost skills: loaded in system prompt but never invoked
  • Repeated file reads within a session
Pass 2
LLM pattern analysis
Batch signal summaries from Pass 1 are sent to an LLM. The LLM identifies cross-session patterns and drafts interventions.
  • Clusters recurring correction types
  • Diagnoses why a skill is underperforming
  • Drafts CLAUDE.md rules and skill rewrites
  • CLI: bring your own API key
  • Team tier: hosted, no key needed

Pass 1 is where SkillCI starts. You can get actionable metrics — reprompt rates, token trends, tool error spikes — from plain JSONL parsing with no external dependencies. LLM analysis layers on top for root-cause reasoning.

Parsing logs yourself

SkillCI ships a reference script at scripts/parse_claude_logs.py that outputs session-level metrics as JSON. It requires only Python 3.9+ from the standard library.

Quick start

# Scan all projects, pretty-print summary
python scripts/parse_claude_logs.py

# Single project directory
python scripts/parse_claude_logs.py --project ~/.claude/projects/-Users-you-code-my-service

# Write to file
python scripts/parse_claude_logs.py --out metrics.json

# Filter to sessions newer than N days
python scripts/parse_claude_logs.py --days 7

Key metric definitions

MetricHow it's computed
reprompt_count_proxy User turns that directly follow another user turn (no assistant turn between them) — a proxy for the user correcting or nudging the agent
tool_error_rate tool_result records where isError is true, divided by total tool calls
compaction_events Count of summary-type records — each one marks where context pressure forced summarisation
cache_hit_ratio cache_read_input_tokens / total_input_tokens across all assistant turns
skills_in_system_prompt Count of SKILL.md or .claude/skills/ references found in the system record

What to look for

High reprompt proxy (>3) — the agent is missing context it needs. Check whether a skill or CLAUDE.md rule addresses the task type.

High tool error rate (>20%) — the agent is exploring unfamiliar territory. A skill describing project structure or file conventions is likely missing.

Multiple compaction events — sessions are hitting context limits. Consider breaking tasks into smaller units or using more focused skills.

Low cache hit ratio (<40%) — the system prompt changes frequently between turns, reducing cache efficiency. Stable CLAUDE.md and skill content improves this.

Skills loaded but reprompts still high — the skills are present but not helping. Pass 2 (LLM analysis) is particularly useful here: it can read the skill descriptions and the correction patterns and identify the mismatch.
